Phase 1 — Audit the Current Setup
Document everything. Change nothing. Client keeps selling on swiftPOS the whole time.
Read-only

1 Record swiftPOS's current configuration No impact

Before anything else, capture what "working" looks like today so you have a rollback reference and can prove nothing broke.

  • Screenshot the swiftPOS main menu (version 10.3.1, build 901-1543-19.09.27, data path D:\SwiftPOS_Elite\Data — already visible in the title bar).
  • Open Utilities and note any hardware configuration screens (printer name, port, drawer settings) swiftPOS itself uses — this tells you the actual working driver/port config for this exact printer and scanner.
  • Confirm who the current manager/login users are (seen: SALIM) and how shifts/reports currently close — you'll need to know their existing end-of-day routine so Elite's shift/Z-report flow doesn't surprise them.
  • Ask the client (or check swiftPOS reports) roughly how many transactions/day and what hours the register is unattended, to size your eventual cutover window.
Why: If Phase 3 cutover ever needs a rollback, you need to know exactly what "before" looked like. This step also tells you the real driver/port settings this specific printer needs, which saves you guessing in Phase 2.

2 Physically trace and label every cable No impact

Follow each cable by hand rather than assuming from the photos.

  • Scanner → confirm it plugs into a USB port on the terminal (not a separate hub/dongle that could be a proprietary interface).
  • Bixolon printer → confirm its connection type: USB, serial, or network/Ethernet. This changes the entire driver approach in Phase 2.
  • Cash drawer → confirm the drawer cable runs from the printer's drawer-kick port, not from the terminal directly. (Photo 3 shows the printer with what could be a modem/handset next to it — visually confirm the drawer separately; it doesn't appear connected in the photo provided.)
  • QNB payment terminal → confirm it is fully independent (its own network/phone line), not looped through the POS terminal at all.
Why: Elite's architecture assumes the drawer is wired to the printer, never the terminal directly (see Hardware Runbook §7). If this install differs, that changes your Phase 2 test plan and Phase 3 rewiring steps.

3 Identify exact hardware models and firmware No impact

  • Bixolon: full model number (e.g. SRP-350plusIII) and firmware version — check a label on the underside/back, or Windows' installed printer driver properties.
  • Scanner: model/serial (there's a maintenance label visible in your photo, but it's obscured — check for a manufacturer name and model on the underside).
  • Terminal: confirm Windows version/edition (visible taskbar suggests Windows 10) and available USB ports/free ports for testing.
Why: The Hardware Runbook explicitly warns not to assume two printer models with similar names share cutter/QR/drawer behavior — you must certify the exact model, not "a Bixolon."
Phase 1 exit criteria: you have photos/notes of swiftPOS's working hardware config, confirmed cable routing, and exact model numbers for printer + scanner — and swiftPOS has not been touched, stopped, or reconfigured at any point.

Phase 2 — Test Elite POS Against This Hardware in Isolation
Your own laptop or a spare machine. Same scanner/printer, borrowed briefly. swiftPOS keeps running.
Zero contact with live install
Key constraint: a USB scanner and a USB/network printer can only be actively claimed by one machine at a time. To test them against Elite without disrupting swiftPOS, you either (a) borrow the scanner/printer's cable for a few minutes during a quiet moment and plug into your own laptop, or (b) if the printer is network-connected, you can test print jobs to it from your laptop over the network without ever unplugging anything, while swiftPOS still owns its own USB scanner connection separately. Confirm which situation you're in during Phase 1, Step 2.

4 Set up an isolated Elite POS test session Your own machine

  1. Bring a laptop (or use a spare machine on-site) with Chrome/Edge installed.
  2. Log in to Elite admin as owner/admin, open /pos, and enroll a test register with an obviously temporary name like TEST-Onsite-Visit — never reuse or overwrite a production register identity.
  3. Open a test shift with a nominal opening float on a test tenant or sandboxed product catalog if available, so any test sales don't pollute real sales/inventory reporting.
Why: This confirms Elite's own software side works before you introduce hardware variables, and guarantees nothing you do here can appear in the client's real sales/inventory numbers.

5 Test the barcode scanner against Elite Borrowed briefly

  1. Unplug the scanner from the Posiflex terminal, plug it into your test laptop. (swiftPOS will simply show no scanner connected until you plug it back — no data loss, no config change.)
  2. In Elite /pos, focus the barcode input field and scan a real product barcode from the client's catalog.
  3. Confirm the scan appends Enter automatically and one exact variant is added to the cart (Hardware Runbook §8).
  4. Scan an unknown/random barcode and confirm the cart does not change.
  5. Scan the same code 10 times quickly and confirm consistent results.
  6. Reconnect the scanner to the Posiflex terminal immediately after testing and confirm swiftPOS sees it again before you leave that station.
Why: this is the exact acceptance test in Hardware Runbook §13.G — passing it here means the scanner model itself is proven compatible before it's ever wired into a live register.

6 Test the Bixolon printer against Elite Borrowed briefly / network test

  1. If USB: briefly disconnect from the Posiflex terminal, connect to your test laptop, and install the vendor Windows driver fresh (don't assume your laptop already has it).
  2. If network-connected: no need to disconnect anything — add the printer's IP as a queue on your laptop directly.
  3. Create a stable queue name, set 80mm paper width, and disable any "convert to graphic" transformation mode so raw ESC/POS passes through (Hardware Runbook §6).
  4. Install and start QZ Tray 2.2.x on your test laptop, confirm the printer queue appears with the exact name.
  5. Complete a test cash sale in Elite's sandboxed session and confirm: receipt prints once, in readable ESC/POS (not garbled), with correct 80mm formatting.
  6. If a drawer is wired to this printer, confirm it pulses once on the cash sale and confirm Pin 2 works before trying Pin 5.
  7. Reconnect everything to the Posiflex terminal exactly as found.
Why: This is the highest-risk unknown from Phase 1 — printer driver quirks are model-specific. Proving this now means Phase 3 has no surprises on the one component most likely to misbehave.

7 Confirm offline behavior works on this hardware Sandbox only

  1. With the test register still connected to scanner + printer, disconnect your test laptop's network/WiFi entirely.
  2. Complete a sandbox sale and confirm it queues locally, gets a reserved receipt number, and still prints via the local QZ signer (Hardware Runbook §10, §13.E).
  3. Reconnect network and confirm the queued sale syncs and the queue count returns to zero.
Why: offline resilience is one of Elite POS's core selling points over swiftPOS — proving it works on this exact hardware combination before cutover avoids discovering a gap on day one of go-live.
Phase 2 exit criteria: scanner and printer both pass their individual acceptance tests against Elite POS on a throwaway test register/tenant. swiftPOS was briefly interrupted only for cable-swap moments (ideally during off-hours) and was fully restored afterward. No real client data touched Elite.

Phase 3 — Scheduled Cutover
Only after Phase 2 passes. Requires a pre-agreed window and a rollback plan.
Live-system change
Do not start Phase 3 the same day as Phase 1/2 unless the client has explicitly approved combining them. Cutover should happen during a scheduled closed/low-traffic window, with the client aware that swiftPOS will be unavailable during the switch and a rollback path is agreed in advance.

8 Agree the cutover window and rollback plan Scheduling

  • Pick a closed hour or lowest-traffic period based on the transaction volume you learned in Phase 1.
  • Confirm with the client: "if anything goes wrong, we revert to swiftPOS and try again another day" — get this agreed verbally/in writing before you start.
  • Do not uninstall swiftPOS during this cutover. Leave it installed and untouched on the machine as the rollback path until Elite POS has been running successfully in production for an agreed trial period (recommend at least one full week of real trading days).

9 Wire the terminal for Elite POS Live change

  1. Reconnect scanner and printer to the Posiflex terminal permanently (they were already tested against this exact terminal's ports in Phase 2 if possible, or an identical port type).
  2. Install the printer driver, QZ Tray, and the Elite offline device signer per Hardware Runbook §5–§10, using the production Elite origin and real signing certificates (not test ones).
  3. Follow Hardware Runbook §11–§12 to enroll this specific register under its real, permanent name (e.g. Main Counter 1) and configure hardware settings in Elite.

10 Run the full acceptance suite before letting the client sell live Required gate

Do not release the register for real trading until every applicable test in Hardware Runbook §13 passes: online cash sale, online card sale, refund receipt, printer failure safety, offline sale and print, restart recovery, and scanner reliability.

11 Run a parallel/trial period before fully retiring swiftPOS Business decision

  • Let the client run real sales on Elite POS for an agreed trial window while swiftPOS stays installed but idle.
  • Check in daily during the trial: any print failures, drawer issues, offline sync surprises, or workflow confusion for the cashier.
  • Only decommission/uninstall swiftPOS once the client explicitly signs off that Elite POS is trusted for daily use.
Why: the biggest real-world migration risk isn't the software, it's a live business losing the ability to sell. Keeping the old system as a safety net for a defined trial period costs nothing and removes essentially all downside risk.
