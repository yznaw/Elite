2
What Needs to Be Built
The concrete gaps between the admin editor and the live website
🔗
Admin → Client connection is missing
Critical gap

The client-web pages (Home, Story, Contact) don't read from StorefrontService. Even if you edit and publish from admin, the website shows the same hardcoded content.

  • HomeComponent never calls StorefrontService
  • StoryComponent has 4 hardcoded chapters in .ts
  • ContactComponent has hardcoded info blocks via i18n
📄
No page scope in block model
Needs design

Currently every block belongs to one pool — the homepage. There's no concept of "this block is for the Story page" or "this block is for Contact". Adding a page field to StorefrontBlock is the minimal fix.

📖
Story page has zero admin controls
Missing feature

The four chapters (1962, 1978, 1995, Today), the hero image, the founder pull-quote, and all six artisan profiles are completely hardcoded in story.component.ts with Unsplash image URLs. They need to be editable from admin.

📬
Contact page info is locked in i18n
Needs admin controls

The three info blocks (Atelier address, Appointments phone/hours, Client Services email) live in i18n translation files. Changing them requires a code deploy. They should be editable in admin without any code changes.

🗂️
Admin UI has one page tab only
UX addition

The Storefront admin page has no concept of "which page am I editing." Adding three tabs — Home, Our Story, Contact Us — makes the scope clear and keeps block lists manageable.

💾
StorefrontService is localStorage-only
Technical debt

The service saves to localStorage with a setTimeout simulating an API call. Express routes for storefront already exist (/api/admin/storefront/*) but aren't wired up yet. The switch to real API calls can be phased.

Data Model Changes
Minimal changes to StorefrontBlock and StorefrontService to support three pages
📦
StorefrontBlock — add page field
interface StorefrontBlock {
  id:      string;
  page:    'home' | 'story' | 'contact';
  type:    string;
  title:   string;
  visible: boolean;
  config:  string;   // notes / icon / attribution

  // Type-specific optional fields (unchanged)
  subtitle?:     string;
  ctaText?:      string;
  ctaLink?:      string;
  imageUrl?:     string;
  productIds?:   string[];
  collectionId?: string;
  itemLimit?:    number;
  sortBy?:       'newest'|'bestseller'|'price-asc'|'price-desc';
  body?:         string;  // also used for JSON payloads
}

The page field is the only required addition. Existing blocks are migrated by defaulting page: 'home'.

⚙
StorefrontService — page-scoped storage
type PageScope = 'home' | 'story' | 'contact';

// localStorage keys become page-scoped:
// 'sf_draft_home'
// 'sf_draft_story'
// 'sf_draft_contact'
// 'sf_published_home'  etc.

interface StorefrontService {
  draft(page: PageScope): Snapshot | null;
  published(page: PageScope): Snapshot | null;
  saveDraft(page: PageScope, blocks: StorefrontBlock[]): void;
  publish(page: PageScope): void;
  revertPublished(page: PageScope, prev: Snapshot): void;
  hasUnpublishedChanges(page: PageScope): boolean;
  buildPreviewLink(page: PageScope): string;
}
🗄
DB snapshots table — add page column
-- Existing table: storefront_snapshots
-- Add one column:
ALTER TABLE storefront_snapshots
  ADD COLUMN page VARCHAR(20)
    NOT NULL DEFAULT 'home'
    CHECK (page IN ('home','story','contact'));

-- Update unique constraint:
-- was: UNIQUE(tenant_id, status)
-- becomes: UNIQUE(tenant_id, page, status)

This migration is safe — no data loss, existing rows default to 'home'.

7
Client-Web Changes
How each customer-facing page reads from the admin-published snapshot
🏠
HomeComponent

What changes:

  • Inject a shared ClientStorefrontService (reads from localStorage or GET /api/storefront/published?page=home)
  • Hero title/subtitle/CTA pulled from the Hero Banner block
  • Featured products carousel filtered to productIds from Featured Products block
  • Brand Story section shows body from Brand Story block
  • New Arrivals section uses itemLimit and sortBy from block config

Fallback: If no published snapshot exists, keep current hardcoded defaults. Never show a blank page.

The THREE.js 3D model viewer stays untouched — it's not admin-controlled.

📖
StoryComponent

What changes:

  • Fetch published 'story' snapshot on ngOnInit
  • Story Hero block → heroImage signal and hero title/tagline
  • Story Chapter blocks → replace hardcoded chapters[] array (year = block.subtitle, title = block.title, body = block.body, image = block.imageUrl)
  • Pull Quote block → replaces hardcoded founder quote
  • Artisan Grid block → parse body JSON → replace hardcoded masters[]

Fallback: if no snapshot, use the current hardcoded arrays as defaults.

📬
ContactComponent

What changes:

  • Fetch published 'contact' snapshot on init
  • Contact Hero block → page headline and intro paragraph
  • Contact Info Block instances → render info cards dynamically (no more i18n strings for address/phone/email)
  • Form Config block → parse subject options from body JSON → populates subject dropdown

Fallback: if no snapshot, existing i18n strings remain as defaults. Zero disruption.

Shared ClientStorefrontService (new — client-web)
@Injectable({ providedIn: 'root' })
export class ClientStorefrontService {

  /** Phase 1: reads from localStorage (same key as admin). */
  /** Phase 2: swap to HttpClient.get('/api/storefront/published?page=…') */

  getPublished(page: 'home' | 'story' | 'contact'): StorefrontBlock[] {
    const raw = localStorage.getItem(`sf_published_${page}`);
    if (!raw) return [];
    try {
      const snapshot = JSON.parse(raw) as { blocks: StorefrontBlock[] };
      return snapshot.blocks.filter(b => b.visible);
    } catch { return []; }
  }

  blockOf(blocks: StorefrontBlock[], type: string): StorefrontBlock | null {
    return blocks.find(b => b.type === type) ?? null;
  }

  blocksOf(blocks: StorefrontBlock[], type: string): StorefrontBlock[] {
    return blocks.filter(b => b.type === type);
  }
}

Phase 1 works immediately since admin already writes to localStorage. The client-web just starts reading from the same keys. Phase 2 switches to a real HTTP call with zero interface changes.
