ESC/POS Thermal Printing — Bixolon

  • 🖨 ESC/POS Direct Print — receipt sent directly to Bixolon 80mm thermal printer via WebUSB (USB-connected) or network socket (LAN-connected). No browser print dialog — one tap, instant print.
  • 💰 Cash Drawer Trigger — after printing, an ESC/POS pulse command is sent through the Bixolon's RJ12 kick port to automatically open the cash drawer.
  • 📧 Email — prompts for customer email if not linked, sends via /api/pos/receipts/:id/email
  • 🌐 Browser Fallback — if WebUSB is unavailable, @media print CSS targets 80mm roll width as a fallback
  • ✕ New Sale — clears cart and starts fresh order
Library: escpos-buffer or node-escpos on the server side generates raw ESC/POS byte streams. The Angular client calls POST /api/pos/print and the server sends bytes to the printer via a local network socket or passes them to WebUSB from the browser. 
// ESC/POS cash drawer trigger (RJ12 via Bixolon)
// Sent immediately after receipt print job
const CASH_DRAWER_PULSE = Buffer.from([
  0x1B, 0x70,  // ESC p — kick-port command
  0x00,        // pin 2
  0x19, 0xFA   // pulse on/off times
]);

05 — X Report & Z Report — Financial Reporting

Industry-Standard X & Z Reports

These are the standard financial reconciliation reports required by retail accountants and store managers worldwide.

📊 X Report — Mid-Shift Read

Run at any point during the shift. Shows current totals without resetting the register. Used by managers to check progress without closing the shift.

  • Total gross sales so far
  • Transaction count
  • Cash takings this shift
  • Card takings this shift
  • Total discounts applied
  • Total refunds processed
  • Net sales (gross minus refunds/discounts)
🔐 Z Report — End-of-Day Close

Run once at end of day. Resets the register counters to zero. The report is stored on the server as a signed, immutable record and printed. Manager PIN required.

  • All X Report data (final for the day)
  • Opening cash float entered at shift start
  • Expected cash in drawer (float + cash sales − change given)
  • Cashier enters physical cash count
  • Over/short variance displayed clearly
  • Report signed with cashier ID + timestamp
  • Stored as immutable record in pos_z_reports table
  • Printed on Bixolon thermal + server copy saved

06 — Barcode Scanner & Label Printing

USB Barcode Scanner

Works as a USB HID keyboard device — no drivers, no config. The search input listens for rapid keystrokes followed by an Enter key.

  • Scanner types ≥ 6 characters in < 100ms → auto-detected as a scan, not manual typing
  • Triggers scanProduct(barcode) on the POS service
  • Product is looked up by product_variants.barcode field
  • If found → item added to cart with a success toast
  • If not found → error toast with the scanned code shown
  • Supports EAN-13, EAN-8, Code 128, QR Code
// Scan detection logic in POS service
let scanBuffer = '';
let lastKeyTime = 0;

document.addEventListener('keydown', (e) => {
  const now = Date.now();
  if (now - lastKeyTime < 80) {
    scanBuffer += e.key;
  } else {
    scanBuffer = e.key;
  }
  lastKeyTime = now;
  if (e.key === 'Enter' && scanBuffer.length >= 6) {
    posService.scanProduct(scanBuffer.slice(0, -1));
    scanBuffer = '';
  }
});

Camera Scanner (ZXing)

Tap the "📷 Camera Scan" button to open a live viewfinder overlay that uses the device camera.

  • Shows live camera feed with scanning reticle overlay

  • Auto-closes viewfinder and adds item to cart
  • Works on iOS Safari, Android Chrome, desktop Chrome/Firefox
  • Requests facingMode: 'environment' (rear camera)

Barcode Label Generation & Printing

Products without an existing barcode can have one generated and printed as a 30×20mm adhesive label directly from the POS or from the Catalog page.

  • Format: Code 128 (alphanumeric SKU) or EAN-13 (numeric variant ID) — auto-selected based on SKU content
  • Label size: 30×20mm — compatible with Dymo LabelWriter 450 and Zebra ZD220
  • Content: Barcode + product name (truncated) + variant (color/size) + price in QAR
  • Generation: bwip-js library generates barcode SVG client-side; label layout rendered to canvas and printed via browser or WebUSB
  • Bulk print: Select multiple variants in the Catalog page → "Print Labels" → specify quantity per variant → batch job
  • Barcode value is saved to product_variants.barcode for future scan lookup

07 — Offline PWA & Park / Resume

Offline-First Service Workers

The POS is built as a Progressive Web App (PWA). If the store's internet drops, the cashier never loses a sale.

  • Service Worker caches the Angular app shell, product catalog, and prices on first load
  • When offline → sales are queued to IndexedDB with a "pending sync" flag
  • Offline queue persists across page refreshes and browser restarts
  • When connection restores → background sync posts queued transactions to /api/pos/transactions automatically
  • Inventory is decremented optimistically locally; reconciled on sync
  • A persistent banner shows online/offline status and pending transaction count
  • Product catalog is refreshed from the server every 15 minutes when online
// IndexedDB queue structure (Dexie.js)
interface PendingTransaction {
  localId:   string;       // UUID generated offline
  items:     CartItem[];
  payment:   PaymentData;
  createdAt: number;       // Unix timestamp
  synced:    boolean;
}

Park & Resume — Multi-Session Cart

A cashier can suspend the current transaction mid-way, serve a different customer, then return to the parked order without losing anything.

  • Up to 5 parked carts can exist simultaneously
  • [⏸ Park Order] button saves current cart to IndexedDB with a name/number
  • A persistent "Parked Orders" tab in the POS topbar shows count
  • Tapping a parked order instantly restores it as the active cart
  • Parked carts expire after 4 hours if not resumed
  • Parked orders are local-only; they do not consume inventory until completed

09 — Payment Gateways — Sadad / QPay / NAPS

Local Qatari Payment Integration

Card payments are routed through local Qatari acquirers — no international gateway fees, no foreign data residency concerns.

🏦 Sadad Payment Solutions
  • Qatar's national electronic payment network
  • Bluetooth terminal pairing for direct charge initiation
  • Receives approval/decline callback from terminal
  • Supports QNB, CBQ, Doha Bank, Al Khaliji cards
💳 QPay / NAPS
  • Qatar's interbank network — VISA, Mastercard, NAPS debit
  • Standalone terminal approach (v1: manual confirm)
  • Phase 2: Bluetooth serial integration for auto-confirm
  • Supports tap (contactless), chip+PIN, swipe
v1 Implementation (Manual Confirm): The POS displays the charge amount and shows "Present card to terminal." The cashier processes on the physical terminal. Once approved, the cashier taps "Confirm Paid" in the POS — this records the card transaction and triggers the receipt. This approach works with any terminal immediately without API integration.

v2 (Future — Direct Integration): Once the acquirer's SDK/API is approved (typically requires a merchant agreement with the bank), the POS will send the charge request directly to the terminal via Bluetooth or local network and receive the approval code automatically.
10 — Returns & Refund System

Refund Flow

A Manager PIN is required to initiate any refund. Two ways to look up the original sale:

  • 📷 Scan original receipt — QR code on receipt encodes the transaction ID, instantly loads the sale
  • 🔍 Search by Order ID or phone number — manual lookup for lost receipts
  • Cashier selects which items to refund and the quantity (partial refunds supported)
  • Refund method must match original payment (cash → cash, card → card)
  • For card refunds: cashier processes reversal on the physical terminal, then confirms in POS
  • Inventory is restored on refund completion
  • Refund creates a negative pos_transaction record linked to the original
  • All refunds appear in X/Z reports as negative line items

Void vs Refund

  • Void — cancels a transaction before it is closed. Manager PIN required. No receipt issued. Inventory restored immediately.
  • Refund — reverses a completed transaction. Tracked separately in reports. Receipt printed with REFUND watermark.
11 — Role-Based Security — Manager PIN

Cashier vs Manager Permissions

The POS enforces two roles. A logged-in admin user can be designated as Cashier or Manager. Certain sensitive operations require a Manager PIN override — entered at the POS, no need to log out.

ActionCashierManagerOverride Method
Open a new sale✔ Allowed✔ Allowed—

Apply discount ≤ 10%✔ Allowed✔ Allowed—
Apply discount > 10%✗ Blocked✔ AllowedManager PIN popup

Manager PIN Override: When a cashier attempts a restricted action, a PIN dialog slides in. The manager enters their 4-digit PIN without logging the cashier out. The action is recorded in the audit log with both the cashier's and manager's IDs. PINs are stored as bcrypt hashes — never plain text.

-- POS Transactions
CREATE TABLE IF NOT EXISTS pos_transactions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  cashier_id      UUID NOT NULL REFERENCES admin_users(id),
  customer_id     UUID REFERENCES customers(id),
  subtotal        NUMERIC(12,2) NOT NULL,
  discount        NUMERIC(12,2) NOT NULL DEFAULT 0,
  total           NUMERIC(12,2) NOT NULL,
  payment_method  VARCHAR(16) NOT NULL,  -- cash | card | split
  cash_amount     NUMERIC(12,2),
  card_amount     NUMERIC(12,2),
  paid_amount     NUMERIC(12,2),
  change_given    NUMERIC(12,2),
  receipt_emailed BOOLEAN DEFAULT false,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- POS Transaction Line Items
CREATE TABLE IF NOT EXISTS pos_transaction_items (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  transaction_id   UUID NOT NULL REFERENCES pos_transactions(id),
  product_id       UUID REFERENCES products(id),
  variant_id       UUID REFERENCES product_variants(id),
  name_snapshot    TEXT NOT NULL,   -- product name at time of sale
  sku_snapshot     VARCHAR(64),
  price_snapshot   NUMERIC(12,2) NOT NULL,
  qty              INT NOT NULL DEFAULT 1
);

CREATE INDEX idx_pos_tx_tenant_date
  ON pos_transactions (tenant_id, created_at DESC);
CREATE INDEX idx_pos_tx_cashier
  ON pos_transactions (cashier_id, created_at DESC);

-- Z Reports (immutable end-of-day records)
CREATE TABLE IF NOT EXISTS pos_z_reports (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  cashier_id      UUID NOT NULL REFERENCES admin_users(id),
  report_date     DATE NOT NULL,
  gross_sales     NUMERIC(12,2) NOT NULL,
  total_refunds   NUMERIC(12,2) NOT NULL DEFAULT 0,
  net_sales       NUMERIC(12,2) NOT NULL,
  cash_takings    NUMERIC(12,2) NOT NULL DEFAULT 0,
  card_takings    NUMERIC(12,2) NOT NULL DEFAULT 0,
  opening_float   NUMERIC(12,2) NOT NULL DEFAULT 0,
  expected_cash   NUMERIC(12,2) NOT NULL,
  physical_cash   NUMERIC(12,2) NOT NULL,
  variance        NUMERIC(12,2) GENERATED ALWAYS AS (physical_cash - expected_cash) STORED,
  tx_count        INT NOT NULL DEFAULT 0,
  closed_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Parked carts (server-side persistence for multi-device)
CREATE TABLE IF NOT EXISTS pos_parked_carts (

  tenant_id   UUID NOT NULL REFERENCES tenants(id),
  cashier_id  UUID NOT NULL REFERENCES admin_users(id),
  label       VARCHAR(64),
  cart_data   JSONB NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ NOT NULL DEFAULT now() + INTERVAL '4 hours'
);

17 — UX Decisions
DecisionChoiceReasoning
Theme Dark (full-width) Hides admin sidebar; easier on eyes at a bright retail counter; premium look
Product grid 3-col image cards Fast visual recognition; large tap targets for touch; luxury product imagery front and center
Cart panel Fixed right side Always visible; no context switching; cashier and customer can both see it
Scanner input USB-first, camera fallback USB is faster and more reliable at counter; camera is a convenience fallback for phones/tablets
Checkout Modal per payment type Minimizes taps under time pressure; one clear action per screen
Currency QAR — no VAT Qatar has no sales tax; matches existing QAR() utility throughout the platform
Customer link Optional at checkout Not all walk-in sales need a customer record; linking is offered but never blocks completion
Receipt print Browser print + 80mm CSS No third-party PDF library; @media print handles 80mm thermal roll layout natively
Inventory Decrement on sale POST Stock updated atomically in the same transaction as the sale record — no drift
Numpad Virtual numpad on touch On tablets, a large on-screen numpad speeds cash entry without relying on system keyboard
Accessibility: All interactive elements ≥ 44px tap target. POS uses high-contrast dark theme with gold/green status indicators. Keyboard-navigable for desktop cashiers. Arabic layout mirrors correctly in RTL mode — cart panel moves to the left, search to the right.
