Bugs Found & Fixed During This Run
IDBugRoot CauseFix AppliedStatus
BUG-01
CO-04		 WhatsApp default placeholder generates broken wa.me link
Default whatsapp value 9744XXXXXXX has non-digit "X" chars. sanitizePhone() strips them leaving 9744 → invalid https://wa.me/9744.		 Placeholder string used as a real value in default content. sanitizePhone() only keeps digits, stripping the X characters. Changed default whatsapp to "" in server defaults, client model, and social link entry. WhatsApp CTA bar is now hidden until a real number is configured by the admin. ✓ Fixed
BUG-02
EC-03		 Orphaned 3D i18n keys remain in translation strings
catalog.has3d, catalog.no3d, catalog.drawer.noModels.* still present in strings.ts for both EN and AR even though 3D product badges were removed from all templates.		 Keys were left in the translation dictionary when template references were removed. Orphaned keys don't cause errors but create clutter and confusion. Removed all 6 orphaned catalog 3D keys from both EN and AR sections of strings.ts. Media library GLB filter kept — it is a legitimate separate feature. ✓ Fixed

Notes & Observations
IDObservationImpactRecommendation
NOTE-01
EC-04		 Empty text field saved as empty string, not replaced by default
When admin clears a content field and saves, the server stores "". The storefront renders an empty gap rather than a default placeholder.		 Low — admin is unlikely to clear required fields. Storefront does not crash, just shows blank where text should be. Acceptable CMS behaviour — an empty save is intentional. Consider adding required-field validation in the admin UI for critical fields (hero title, story hero title) with a save-time warning, without blocking the save.
NOTE-02
PV-07		 No Arabic product translations in DB yet
product_translations table has 0 rows with locale='ar'. The schema and server code are correct.		 None — data entry has not yet been done. Feature is ready. Admin should enter Arabic names when editing products. Bulk import CSV template should include a nameAr column example.
NOTE-03
HS-04/05/06		 Add/remove slide & callout tests require browser
These tests require the admin UI to be open in a browser. The underlying API layer (PATCH with new items array) was verified to work end-to-end.		 None — API layer confirmed working. UI interaction not automatable from CLI. Test manually in browser: open admin → Landing Hero → Add new slide → fill fields → Save Content → verify storefront carousel includes new slide.
NOTE-04
CT-03		 Collection cover image requires SEO imageUrl field
The live collection tile join reads seo->>'imageUrl' from the collections table. The cover image uploaded via "Replace cover" uses cover_media_id, which is NOT read by the storefront tile.		 Medium — admin may upload a cover via the media picker expecting it to appear on the storefront tile, but only the SEO imageUrl field is synced. Either: (1) join media_assets on cover_media_id to get the cover URL, or (2) make admin aware that "Use image URL" (SEO field) is what the storefront tile uses.
