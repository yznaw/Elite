System Integration Map — Read Before Coding
Every touchpoint between Reference data, Variants, Images, and the Storefront
▼
The chain: ref_colors → product_variants.color (text FK, unforced) → media_assets.metadata.color (JSON) → imageColors map in admin form → colorImages in public API → storefront swatch → image gallery swap.

Every phase below must keep this chain intact. A color rename in ref_colors currently does NOT propagate to variant rows — that is a known gap addressed in Phase 2.
⚠ Leather & Footwear Color Considerations
Leather goods present unique color challenges not seen in apparel:
• Finish variation — "Camel Smooth" vs "Camel Pebbled" are the same hex but different texture. The hex alone is insufficient for product photography matching.
• Aging/patina — full-grain leather darkens with use; product images show new leather, not aged. Color swatches should show "new" state.
• Lighting sensitivity — a single leather color can look tan, brown, or golden under different lighting. Use a neutral-lit reference photo as the swatch source, not just a CSS hex.
• Exotic leathers (crocodile, ostrich) have pattern-based appearance, not solid color. These need a thumbnail swatch image, not a flat hex circle.
• Bi-color products — sandal strap + sole may be different colors. The variant model (one color per variant) covers this only if the "primary" color convention is documented.
Solution in this plan: Phase 3 adds an optional swatch_image_url field to ref_colors — when set it overrides the flat hex circle in swatches everywhere.

Migration 010 — Color-Image Pivot + Swatch Image
Replace fragile JSON metadata color linking with a proper relational table
DB Pending
0%
▼
Currently media_assets.metadata->>'color' (a JSON blob field) is the only link between an image and a color. It has no FK, no cascade, no integrity. This migration adds a proper pivot table and an optional swatch image field so exotic leathers can display a texture thumbnail instead of a flat hex circle.
⚡ Zero-downtime strategy: New table is additive. During Phase 2 the API writes to BOTH the old metadata path AND the new pivot. Rollback = drop the migration. The old metadata path stays working until Phase 7 cleanup.

Core insight: A product with 4 colors × 5 sizes has 20 variant rows today. The admin must click the photo cell on each of the 5 "Camel" rows to assign the same image — 5 redundant actions. The grouped layout reduces that to 1 action per color. The save payload { variants[], imageColors{} } does NOT change — grouping is purely a view transformation computed from the flat array.

The fix: compute availableSizes as a derived signal that depends on selectedColor(). Sizes not existing for the selected color show as greyed-out (not hidden — this is the Shopify/WooCommerce standard: hide nothing, grey what's unavailable).
Leather PDP Color UX Note
Luxury leather customers expect to see ALL colorways upfront (not hidden on a dropdown). The swatch strip below the product title is the primary color navigation. Selecting a color should:
1. Swap the hero image to that color's photo
2. Filter sizes to show only what exists in that color
3. Grey-out sizes with stock=0 in that color (not remove them — "coming back" signal)
4. Update the URL: ?color=camel for sharing
