Critical Frontend
Bug #1 — Customer drawer shows MOCK orders, not real API data
File: pages/customers/customer-drawer.component.ts — Line 17 + Lines 392-396
import { ORDERS } from '../../data/mock';
readonly customerOrders = computed<Order[]>(() => ORDERS.filter(o => o.customer === name));

The customer history section in the drawer — order rows, size-preference bar chart, order count — is powered entirely by hardcoded mock data. Even if an admin scrolls into a real customer's drawer, every order shown is fake. New real orders from the storefront will never appear.

Root cause: The GET /admin/customers/:id endpoint exists but doesn't return orders. There is no GET /admin/customers/:id/orders endpoint. The developer temporarily wired mock data and never came back to fix it.

Fix required (2 parts):

  • Backend: Add GET /admin/customers/:id/orders — returns all orders where customer_id = :id OR customer_email = customer.email
  • Frontend: In CustomerDrawerComponent.ngOnInit(), call the new endpoint and replace the mock computed. Remove the mock import entirely.
Critical Frontend
Bug #2 — Customer save shows "Saved" before the API call resolves
File: pages/customers/customer-drawer.component.ts — save() method (Lines 342-361)
The method uses a hard-coded setTimeout(..., 700) and immediately mutates this.customer in memory, showing the "Saved" toast — before saved.emit() ever triggers the actual this.customersApi.update() call in the parent.

If the API call in the parent fails (network timeout, server error), the drawer has already shown "Saved" and the customer object has already been mutated in memory. The admin believes the data was persisted when it wasn't.

Fix: Move the save logic entirely into the drawer component using AdminCustomersService directly. Remove the setTimeout fake delay. Only set saveState = 'saved' and emit after a successful API response. Keep saveState = 'saving' while the call is in-flight and show a real spinner.

High Backend
Bug #3 — Hard-deleting customers orphans their orders
File: server/routes/admin-customers.route.js — DELETE /:id
DELETE FROM customers WHERE tenant_id = $1 AND id = $2

When a customer is deleted, all orders that have customer_id = deleted_id retain the now-orphaned FK. The orders remain visible in the orders list (using the denormalized customer_name column), but any attempt to load "this customer's orders" from the customer table will fail silently — and analytics LTV calculations will be wrong.

Fix: Replace hard delete with a soft delete. Add deleted_at TIMESTAMPTZ NULL to the customers table. DELETE API should SET deleted_at = now(). All SELECT queries add WHERE deleted_at IS NULL. If a customer must be fully purged, that's an explicit separate operation that also nullifies customer_id on orders.

Critical Backend
Add GET /admin/customers/:id/orders

Returns the full order history for a specific customer. Must join by both customer_id AND by email match to handle orders placed before the customer record existed (guest checkout, storefront orders that pre-date the admin record).

router.get('/:id/orders', asyncHandler(async (req, res) => {
  const client = await db.pool.connect();
  try {
    const tenant = await ensureDefaultTenant(client);
    // find customer first (email is needed for fallback join)
    const cust = await client.query(
      'SELECT id, email FROM customers WHERE tenant_id = $1 AND id = $2',
      [tenant.id, req.params.id]
    );
    if (cust.rowCount === 0) return notFound(res, 'Customer not found.');
    const { id: custId, email } = cust.rows[0];

    const result = await client.query(`
      SELECT o.*, COUNT(oi.id)::int AS items_count,
        COALESCE(jsonb_agg(DISTINCT jsonb_build_object(
          'n', oi.product_name, 's', COALESCE(oi.size,''),
          'q', oi.quantity, 'p', round(oi.unit_price_cents/100.0)
        )) FILTER (WHERE oi.id IS NOT NULL), '[]'::jsonb) AS items
      FROM orders o
      LEFT JOIN order_items oi ON oi.order_id = o.id
      WHERE o.tenant_id = $1
        AND (o.customer_id = $2 OR o.customer_email = $3)
      GROUP BY o.id
      ORDER BY o.placed_at DESC
    `, [tenant.id, custId, email]);

    ok(res, result.rows.map(mapOrder));
  } finally { client.release(); }
}));

High Backend
Order creation idempotency — prevent duplicate orders on retry

The current POST /admin/orders generates publicNumber as EC-YY-${Date.now().slice(-5)}. A 5-digit millisecond suffix has a collision window when two requests land within 100ms, or when the client retries a failed request (network timeout after the server already inserted the row).

Two-layer fix:

  • DB layer: Add UNIQUE (tenant_id, public_number) constraint to the orders table. The INSERT will fail with a unique violation rather than creating a silent duplicate.
  • API layer: Accept an optional idempotencyKey in the request body. Before inserting, check SELECT id FROM orders WHERE tenant_id = $1 AND idempotency_key = $2. If found, return the existing order (HTTP 200, not 201). This lets the frontend safely retry without duplicating.
// Frontend: generate before sending
const idempotencyKey = `order-${crypto.randomUUID()}`;
// store in localStorage against the draft, re-use on retry
Also fix: The publicNumber generator should use a longer unique suffix or a sequential counter per tenant. Recommended: EC-YY-MMDD-XXXXX where XXXXX is from a DB sequence (nextval('order_seq')), guaranteed unique without collision.

High DB
Verify v_customer_order_stats view exists and has correct fallback

The GET /admin/customers route depends on v_customer_order_stats via a LEFT JOIN. If this view doesn't exist or has a bug, every customer list fetch fails with a 500 error and the entire customers page shows empty/error.

Required view definition (create if missing):

CREATE OR REPLACE VIEW v_customer_order_stats AS
SELECT
  customer_id,
  COUNT(*)::int AS orders_count,
  COALESCE(SUM(total_cents), 0)::bigint AS ltv_cents,
  MAX(placed_at) AS last_order_at
FROM orders
WHERE customer_id IS NOT NULL
  AND payment_status = 'paid'
GROUP BY customer_id;

Fallback in query: The LEFT JOIN already handles the "no orders" case via COALESCE. Add an ON CONFLICT DO NOTHING guard in the view creation migration so re-running migrations is idempotent.

Medium Backend
NBOX booking: add admin notification on retry-exhausted failure

When bookNboxForPaidOrder fails, the code logs a console warning and adds a timeline entry. The admin has no UI feedback that the delivery was never booked unless they open that specific order and read the timeline.

Short-term: The timeline entry added on failure should use kind: 'note' with a red-flag detail string that the order drawer renders distinctly (e.g. a warning pill). The customer list page's silent refresh should surface any order where the timeline has an unresolved NBOX failure as a badge.

After 3 failures, set a requires_manual_booking flag and surface it in the orders page as a prominent banner.

Edge Cases & Fallback Scenarios
Every scenario where things go wrong — and what the system should do
Order-Related Edge Cases
Critical
Scenario: Customer places duplicate order (double-tap, retry, back-button)

Risk: Customer taps "Place Order" twice on the storefront or the network drops after the server commits but before the success response reaches the client. The client retries. Without idempotency, two identical orders are created.

Client sends POST /orders with idempotency_key
→
Server checks idempotency_key in orders table
→
Key not found → INSERT → return 201
Client retries (network error)
→
Server finds existing idempotency_key
→
Return existing order → 200 (not a new insert)

Admin-side detection: If a duplicate slips through anyway (e.g. two different clients), the admin orders list should group orders by customer and date and highlight any where: same customer + same total + same items within 60 minutes. Show a "Possible Duplicate" badge on the newer one.

Implementation: Add idempotency_key TEXT NULL UNIQUE per tenant column to orders. Storefront generates UUID before checkout starts, persists in localStorage, re-uses on retry. Cleared from localStorage only after confirmed server 201.

High
Scenario: Customer makes a payment but status is stuck on "pending"

Risk: Payment gateway webhook fires but fails (timeout, server restart). The Sadad/Stripe/payment webhook route marks the order as paid, but if it crashes mid-update, the order stays pending forever. The admin sees a pending order but the customer's bank shows a deduction.

Fallbacks:

  • The order drawer should show a "Confirm Payment Manually" button prominently for payment === 'pending' orders — this button already exists but should be visually elevated when the order is more than 30 minutes old.
  • Add a payment reconciliation query: SELECT * FROM orders WHERE payment_status = 'pending' AND placed_at < now() - interval '30 minutes'. Surface this count as a red badge on the orders nav item.
  • Payment gateway should implement webhook retry with exponential backoff. The sadad-webhook.route.js should be idempotent (check if already paid before updating).

In the order drawer: When payment = 'pending' AND date < today - 1 day, show a warning banner: "This order has a pending payment older than 24h. Check payment gateway or confirm manually."

High
Scenario: Status update API call fails mid-transition (optimistic vs real state)

Risk: Admin clicks "Mark Shipped". The drawer shows "Shipped" optimistically. The PATCH request fails with 500. The in-memory order now says "shipped" but the database still says "processing". The admin sees the wrong state until they close and re-open the drawer.

Current behavior: The order drawer does NOT optimistically update. It waits for the API response (this._order.set(updated) runs only after the resolved promise). This is correct — but the error catch only surfaces via the global toast interceptor. The drawer needs to also re-fetch on error to ensure state is correct.

Fix in order-drawer.component.ts:

async transition(target: OrderFulfillment): Promise {
  this.busy.set(true);
  try {
    const updated = await this.ordersApi.updateStatus(...);
    this._order.set(updated);
    this.updated.emit(updated);
    // ... toast
  } catch {
    // Refetch to ensure local state matches server
    try { const fresh = await this.ordersApi.get(o.id); this._order.set(fresh); } catch {}
    // Global interceptor shows the error toast
  } finally { this.busy.set(false); }
}
High
Scenario: Customer creates a new order — it must appear in their customer drawer immediately

Risk: Admin has the customer drawer open. A new order arrives from the storefront. The customer drawer shows the old order list (fetched once on drawer open, never refreshed). Admin doesn't see the new order without closing and re-opening the drawer.

Fix — Customer drawer order refresh:

  • When the customer drawer opens, load orders from GET /admin/customers/:id/orders into a local signal orders = signal<Order[]>([]).
  • Set up a 30-second poll (similar to the orders page 15s refresh) inside the drawer. On each tick, silently re-fetch the customer's orders and update the signal.
  • The orders list in the drawer should show a subtle "Updated just now" timestamp so the admin knows the data is live.
  • Cancel the poll in ngOnDestroy.

This is especially important because: when an admin is processing an order, a customer might call in saying "I just placed another order" — the admin needs to see it without navigating away.

API & Network Edge Cases
High
Scenario: API call fails — add retry with exponential backoff

Currently ApiClient makes a single HTTP request and throws on error. No retry logic exists. A transient 502 or DB connection pool exhaustion will show an error toast and drop the operation. On mobile with intermittent signal this is frequent.

Add an RxJS retry operator to ApiClient:

import { retry, timer } from 'rxjs';

// Inside get/post/patch/delete, pipe before map:
.pipe(
  retry({
    count: 2,                        // max 2 retries (3 total attempts)
    delay: (_, retryIndex) =>
      timer(Math.pow(2, retryIndex) * 500), // 500ms, 1000ms
    resetOnSuccess: true,
  }),
  map(res => res.data),
)

Important: Only retry GET and safe read operations automatically. For mutating requests (POST, PATCH, DELETE), do NOT auto-retry without idempotency keys — retrying a PATCH could cause unintended double-updates. For those, surface the error and let the admin retry manually.

Medium
Scenario: 401 session expiry while admin is mid-workflow

If the Express session expires (default 24h, or server restart), every API call returns 401. Currently the HTTP interceptor surfaces a toast error. But if the admin is in the middle of a status update or saving a note, the work is silently lost — the toast just says "Unauthorized".

Handling in the HTTP interceptor:

  • On 401: show a modal (not a toast) saying "Your session has expired. Log in again to continue." with a "Go to Login" button.
  • Preserve the current URL so after re-login the admin lands back on the same page.
  • If there's an in-progress order drawer with a pending note draft, save it to localStorage before redirect so it can be restored after re-auth.
Medium
Scenario: Orders list returns empty due to v_customer_order_stats view failure

If v_customer_order_stats fails (view dropped, DB migration error), the LEFT JOIN in the customers list query may throw a 500, making the entire customers page show an error state. The orders page is independent and won't be affected, but admins can't access customer data at all.

Fix — Graceful fallback query: Wrap the view join in a try/catch inside the route and fall back to the customers table's own orders_count and ltv_cents columns if the view is unavailable.

try {
  // full join with view
} catch (viewErr) {
  console.warn('v_customer_order_stats unavailable, using denormalized columns', viewErr.message);
  // fallback: SELECT c.* FROM customers c WHERE c.tenant_id = $1
  result = await client.query('SELECT * FROM customers WHERE tenant_id = $1 ORDER BY joined_at DESC', [tenant.id]);
}

High Frontend
Orders page: distinguish "loading" from "empty" from "error" states

Currently when the initial fetch fails, the catch block does this._orders.set([]) and removes the loading spinner. The page shows the empty state ("No orders found") as if there genuinely are no orders. The admin has no idea the data failed to load.

Add an error signal:

readonly loadError = signal<string | null>(null);

private async refreshOrders(silent = false): Promise<void> {
  if (!silent) { this.loading.set(true); this.loadError.set(null); }
  try {
    this._orders.set(await this.ordersApi.list());
  } catch (err: any) {
    if (!silent) {
      this.loadError.set('Failed to load orders. Check your connection.');
      this._orders.set([]);
    }
    // silent refresh failures are swallowed (don't flash errors every 15s)
  } finally {
    if (!silent) this.loading.set(false);
  }
}

In the template: show a red callout banner with a "Retry" button when loadError() is set, instead of the empty state.

Medium Frontend
Order drawer: add a "stale data" refresh guard after errors

When any workflow action (transition, tracking save, cancel, refund) fails, the in-memory _order signal may be out of sync with the DB. The fix (from the edge cases section) is to re-fetch the order on error. This needs to be centralized:

private async safeUpdateStatus(payload: OrderStatusPayload): Promise<void> {
  const o = this._order();
  try {
    const updated = await this.ordersApi.updateStatus(o.id, payload);
    this._order.set(updated);
    this.updated.emit(updated);
  } catch {
    // Re-fetch to sync local state
    try { this._order.set(await this.ordersApi.get(o.id)); } catch {}
    throw; // re-throw so the caller's toast fires
  }
}

All workflow methods (transition, confirmPayment, cancelOrder, refundOrder, saveTracking) call safeUpdateStatus() instead of ordersApi.updateStatus() directly.

Data Flow for Customer Drawer
Critical Frontend
Customer drawer: full data flow rewrite

Replace the current mock-data implementation with a proper async data flow:

// customer-drawer.component.ts

// 1. Add inject
private readonly customersApi = inject(AdminCustomersService);

// 2. Add signals
readonly orders = signal<Order[]>([]);
readonly ordersLoading = signal(false);
readonly ordersError = signal<string | null>(null);
private refreshTimer: ReturnType<typeof setInterval> | null = null;

// 3. ngOnInit
async ngOnInit(): Promise<void> {
  this.initial.set(this.makeFormFromCustomer(this.customer));
  this.form.set({ ...this.initial() });
  if (this.mode === 'edit') {
    await this.loadOrders();
    this.refreshTimer = setInterval(() => void this.loadOrders(true), 30_000);
  }
}

// 4. ngOnDestroy
ngOnDestroy(): void {
  if (this.refreshTimer) clearInterval(this.refreshTimer);
}

// 5. Load orders method
private async loadOrders(silent = false): Promise<void> {
  if (!silent) this.ordersLoading.set(true);
  this.ordersError.set(null);
  try {
    const list = await this.customersApi.getOrders(this.customer.id);
    this.orders.set(list);
  } catch {
    if (!silent) this.ordersError.set('Could not load order history.');
  } finally {
    if (!silent) this.ordersLoading.set(false);
  }
}

// 6. Size stats — derived from REAL orders
readonly sizeStats = computed(() => {
  const sizeMap: Record<number, number> = {};
  this.orders().forEach(o =>
    o.items.forEach(it => { sizeMap[it.s] = (sizeMap[it.s] || 0) + it.q; })
  );
  const sizes = Object.keys(sizeMap).map(s => ({ size: +s, count: sizeMap[+s] }));
  return sizes.length ? sizes.sort((a,b) => b.count - a.count)
    : [{ size: this.form().sizePref || 0, count: 1 }];
});

// Replace the current save() method entirely:
async save(): Promise<void> {
  if (!this.dirty() || this.saveState() === 'saving') return;
  const f = this.form();
  if (!f.name.trim() || !f.email.trim()) {
    this.toast.error('Name and email are required.');
    return;
  }
  this.saveState.set('saving');
  try {
    const saved = this.mode === 'create'
      ? await this.customersApi.create({ name: f.name, email: f.email, city: f.city, sizePref: f.sizePref, notes: f.notes })
      : await this.customersApi.update(this.customer.id, { name: f.name, email: f.email, city: f.city, sizePref: f.sizePref, notes: f.notes });

    // Adopt server response (especially id for new records)
    Object.assign(this.customer, saved);
    this.initial.set({ ...f });
    this.saveState.set('saved');
    const titleKey = this.mode === 'create'
      ? 'customerDrawer.toast.created.title'
      : 'customerDrawer.toast.saved.title';
    this.toast.success(this.t(titleKey), f.name);
    this.saved.emit({ ...this.customer, ...saved });
    setTimeout(() => this.saveState.set('idle'), 1500);
  } catch {
    this.saveState.set('dirty'); // revert to dirty so admin can retry
    // Global interceptor shows the error toast
  }
}

The parent onCustomerSaved should no longer call the API — the drawer handles it directly. The parent only needs to update the list signal when it receives the saved emit.

Orders Page — UX Improvements
Specific UX issues found during the audit, with fixes
Medium UX
Add a "Stale Payment" warning badge to orders with pending payment >30 min

In the orders list, if o.payment === 'pending' AND o.date < today - 1, add a small warning icon next to the pending pill. In the order drawer, show a yellow callout banner at the top of the workflow section.

This gives admins an immediate visual cue without needing to open every pending order to check if it's a stuck payment vs. a new order.

Medium UX
Order drawer: show product thumbnail placeholder linked to real product

The line items section in the order drawer shows a grey placeholder box (background: linear-gradient(135deg, #e8eaf2, #dde1ee)) where a product image should be. The OrderItem model has no image URL — the image needs to come from a join with media_links via product_id.

Backend: In loadAdminOrder(), the items jsonb_agg should include the product's cover image:

jsonb_build_object(
  'n', oi.product_name,
  's', COALESCE(oi.size, ''),
  'q', oi.quantity,
  'p', round(oi.unit_price_cents / 100.0),
  'img', (SELECT url FROM media_links WHERE product_id = oi.product_id AND role = 'cover' LIMIT 1)
)

Frontend model: Add img?: string to OrderItem. In the drawer template, render <img [src]="api.mediaUrl(it.img)"> instead of the grey div.

Medium UX
Order drawer: add "Open Customer Profile" button

Currently in the Customer section of the order drawer, the admin can see the customer name + email + phone but has no way to navigate to the customer's profile without closing the drawer and going to the customers page.

Add: A small "View Customer" ghost button next to the customer name that navigates to /customers?highlight={customerId}. On the customers page, the highlight param scrolls to and opens the customer's drawer automatically.

Low UX
Order search: add debounce on search input (300ms)

The current search uses (ngModelChange)="search.set($event)" which fires a recompute on every keystroke. At 500+ orders this can cause visible lag on mobile. Add a 300ms debounce:

// Convert to rxjs Subject approach or use a debounce signal wrapper
private readonly searchInput$ = new Subject<string>();

ngOnInit() {
  this.searchInput$.pipe(debounceTime(300), takeUntilDestroyed())
    .subscribe(v => { this.search.set(v); this.page.set(0); });
}
